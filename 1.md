1. Channel Type & Declaration

Channels are typed concurrency primitives used for fiber-safe communication.

Declaration
var ch : channel:str = channel();


channel:T is a first-class type

Channels are heap objects, managed by the VM

2. Channel Operations (Language Level)
Blocking operations (fiber-aware)
ch.send(value );     // blocks if full
ch.recv(value from ch);  // blocks if empty


These may suspend the current fiber

Scheduler decides resumption

Never used implicitly in expressions

Non-blocking operations (NEW)
offer — non-blocking send
if (ch.offer(value)) {
    print("sent");
} else {
    print("channel full");
}


Attempts to send

Returns bool

Never blocks

Never suspends fiber

poll — non-blocking receive
if (ch.poll(msg)) {
    print(msg);
} else {
    print("no data");
}


Attempts to receive

Assigns into target variable

Returns bool

Never blocks

Never suspends fiber

3. Why poll / offer (Design Rationale)

Channel empty/full is not exceptional

Avoids misuse of try

Explicit control over blocking

VM-predictable

JIT-safe

Scheduler-agnostic

4. AST NODES (Authoritative)
4.1 ChannelOfferExpr
struct ChannelOfferExpr : Expr {
    Expr* value;          // value being sent
    Expr* channel;        // channel expression
};


Semantic rules:

Returns bool

Does not suspend

Fails immediately if full

4.2 ChannelPollExpr
struct ChannelPollExpr : Expr {
    Expr* channel;        // channel expression
    Identifier target;    // variable to assign
};


Semantic rules:

Returns bool

Assigns target only on success

Does not suspend

5. LIR INSTRUCTIONS (REPLACES OLD MODEL)
❌ REMOVED (old)
ChannelAlloc
ChannelPush
ChannelPop
ChannelHasData


These are deprecated and must not be emitted.

✅ NEW LIR OPS
ChannelCreate    r_ch, capacity
ChannelOffer     r_ch, r_value, r_result
ChannelPoll      r_ch, r_target, r_result

6. LIR SEMANTICS
ChannelCreate
ChannelCreate r1, 2


Allocates channel object

Stores pointer in r1

ChannelOffer
ChannelOffer r_ch, r_val, r_ok


VM behavior:

If buffer has space:

push value

r_ok = true

Else:

r_ok = false

Never suspends

ChannelPoll
ChannelPoll r_ch, r_out, r_ok


VM behavior:

If buffer has data:

pop value

store into r_out

r_ok = true

Else:

r_ok = false

Never suspends

7. Example: End-to-End Translation
Source
var ch : channel:str = channel(capacity=2);

if (offer("hello" to ch)) {
    print("sent");
}

if (poll(msg from ch)) {
    print(msg);
}

AST (simplified)
VarDecl(ch : channel:str)
 └─ ChannelCreate(capacity=2)

If
 └─ ChannelOffer("hello", ch)

If
 └─ ChannelPoll(ch, msg)

LIR
ChannelCreate  r1, 2
LoadConst      r2, "hello"
ChannelOffer   r1, r2, r3
JumpIfFalse    r3, L_skip1
CallPrint      "sent"

ChannelPoll    r1, r4, r5
JumpIfFalse    r5, L_skip2
CallPrint      r4

8. Blocking Still Exists (Explicit Only)

Blocking ops are separate instructions:

ChannelSend   // may suspend fiber
ChannelRecv   // may suspend fiber


Only emitted for send / recv

Never used implicitly

Scheduler interaction required

9. VM & JIT Guarantees

poll / offer are pure VM ops

No fiber state mutation

Safe for libgccjit

Predictable control flow

No hidden yields

10. Final Rule (Important)

Blocking is a statement.
Non-blocking is an expression.

This rule is enforced at:

AST validation

LIR generation

VM execution
